# 管理画面 UI フィールド説明書

この文書は、**Line Summary Console**（`index.html` / Supabase `admin-ui`）および **LINE Media Viewer**（`media.html`）に表示される**主なテーブル・チェックボックス・入力欄**の意味と、バックエンド上の対応関係を整理したものです。  
画面ラベルは実装に合わせた表記です。

---

## 共通前提

| 項目 | 説明 |
|------|------|
| API | 管理 UI は `admin-api`（例: `/functions/v1/admin-api`）に接続します。 |
| 認証 | 管理トークン（`ADMIN_DASHBOARD_TOKEN` 相当）をヘッダ `x-admin-token` で送ります。ブラウザでは LocalStorage に保存されます。 |
| 即時保存 | **ユーザー権限**のチェック変更は、その場で API に送信され即時保存されます。 |
| ルーム設定 | ルーム行の**配信時刻・表示名・最終回集計**などの変更は、一定時間後に自動保存されるほか、手動で再読み込み前に反映されます。ルーム設定モーダルは「保存」で確定します。 |

---

## 1. Line Summary Console — 管理トークン

| UI | 説明 |
|----|------|
| 管理トークン入力 | 接続に使う秘密文字列。保存すると以降の API 呼び出しに使用されます。 |
| 保存して接続 / 削除 | トークンの保存または削除。 |
| 再読み込み | サーバーから最新の全体状態を取得します。 |
| 今すぐ要約実行 | 定期ジョブ相当の要約処理を手動トリガーします（結果は配信ログに反映）。 |
| メディア閲覧 | `media.html`（メディア・資料ライブラリ画面）へのリンク。 |
| Gmail連携先を確認 | 連携中の Gmail アカウント表示の確認用。 |
| 管理トークン変更 | 新しいトークンへ更新（以降は新トークンが必要）。 |

## 2. Line Summary Console — 全体設定

### 2.1 配信設定ブロック

| フィールド | 説明 |
|------------|------|
| **配信を有効化** | グローバルな定期配信（要約など）のマスター ON/OFF。OFF の場合、スケジュールに沿っても配信しない扱いになります。 |
| **配信時刻（例: 12,17,23）** | 1日あたりの配信「時刻」一覧（0〜23 の整数、カンマ区切り）。**時刻の個数＝1日の配信回数**として解釈されます。 |
| **最終配信の集計方式** | **各回独立で要約（従来）**: 配信ごとにその区間の要約。**最終配信のみ1日まとめ**: 1日の最後の配信枠でその日をまとめて要約するモード。 |
| **会話保持** | `line_messages` 等の**保持期間**（日数）。期限超過分はクリーンアップ対象になり得ます。無制限も選択可。 |
| **全体設定を保存** | 上記を `global_settings` に保存。 |

### 2.2 翌日予定通知ブロック

| フィールド | 説明 |
|------------|------|
| **翌日予定通知を有効化** | 翌日のカレンダー予定を LINE で通知する機能の ON/OFF。 |
| **通知時刻** | 翌日予定を送る時刻（カンマ区切り、全体デフォルト）。 |
| **予定なしの日も通知する / 予定がある日だけ** | 空振りメッセージを送るかどうかのポリシー。 |
| **表示件数** | 通知に含める予定の最大件数（上限あり）。 |

### 2.3 メタ情報・ストレージ

| 表示 | 説明 |
|------|------|
| **globalMeta** 付近のテキスト | 配信回数・最終回モード・保持日数など、読み取り専用の要約表示。 |
| **DB テーブル容量（円グラフ）** | Postgres 上のテーブル別おおよその使用量（監視用）。 |

## 4. Line Summary Console — ルーム別設定

### 4.1 画面上部の注意文

管理ポリシーにより、**会話検索応答・資料検索（2段階目）・一部自動応答**は誤操作防止のため、この画面では**チェックボックスとして表示していません**（バックエンド側で固定または別経路で制御）。

### 4.2 新規ルーム追加行

| フィールド | 説明 |
|------------|------|
| **新規 room_id** | LINE のグループ/ルーム識別子。必須。 |
| **表示名** | 管理画面上の名前（任意）。 |
| **時刻** | このルーム専用の配信時刻。**空欄**の場合は**全体設定の配信時刻**を継承。 |
| **追加時の設定** | モーダルを開き、次回「ルーム追加」時に使う**既定のチェック**と**最終回集計**を編集。 |
| **既定サマリー表示** | 例: `3/7 有効 ・ 最終回:継承` — モーダル内で ON になっている機能数と、最終回の集計モードの要約。 |
| **ルーム設定を追加** | 入力に基づき `room_settings` を作成・更新。 |
| **ルーム名再取得** | LINE 側の表示名を取り直して反映する操作。 |

### 4.3 ルーム一覧テーブル（列）

| 列 | 説明 |
|----|------|
| **ID** | ドラッグハンドルで並び替え、`ID表示` でルーム ID をコピー用モーダル表示。 |
| **表示名** | ルーム名の編集欄。下に**最終投稿**日時。 |
| **未処理件数** | 要約・処理待ちなどのメッセージ件数（サーバー値）。 |
| **設定** | `設定` ボタンで**ルーム設定モーダル**。**バッジ**は有効になっているルーム機能の数（例: `4/7 有効`）を色分け表示。 |
| **配信時刻** | ルーム別の上書き時刻。空欄で全体継承。 |
| **最終回集計** | `継承` / `各回独立` / `1日まとめ`。全体の「最終配信の集計方式」との組み合わせに注意。 |
| **操作** | `メンバー同期`（**グループ `C…` / 複数人トーク `R…` の行のみ**）: LINE **メンバー ID 一覧 API** で当該トークの全員を取得し **ユーザー権限**（`line_user_permissions`）へ反映。既存ユーザーは権限を維持して表示名を更新、未登録は Webhook 新規と同様の未承認既定で追加。**注意**: 当該 API は LINE 側の仕様で **認証済みまたはプレミアムの公式アカウント**に限定されており、未認証チャネルでは 403 になります。その場合はメッセージ受信 Webhook や「既存ユーザー取込」でユーザーが取り込まれます。`継承`: ルーム個別設定を削除してデフォルトへ。`ルーム削除`: ルームと関連データの削除（確認あり）。 |

### 4.4 ルーム設定モーダル（チェック一覧）

**「有効」単独のチェックはありません。**  
保存時、次の**いずれか一つでも ON なら**内部的な `is_enabled` が true になります（すべて OFF なら false）。

| チェック | 意味（概要） |
|----------|----------------|
| **AI会話返信** | そのルームで、Bot による**会話・意図判定に基づく返信**を許可するか（`bot_reply_enabled`）。 |
| **ルーム要約配信** | このルームを**定期要約配信の対象**にするか（`send_room_summary`）。 |
| **明日予定配信** | このルーム向けに**翌日予定通知**を送るか（ルーム単位の上書き）。 |
| **LINE添付保存（ルーム）** | 当該ルームで送受信した**画像等のメディアを Storage に保存**するか（`media_file_access_enabled`）。資料ライブラリの「誰が読めるか」とは別です。 |
| **自動登録** | 自然文からの**カレンダー予定の自動登録**を許可するか（`calendar_ai_auto_create_enabled`）。 |
| **無返信即時登録（低確度は仮）** | 返信なしでカレンダーに書き込む**サイレント登録**モード（信頼度に応じ仮タイトル等）。 |
| **Gmail予約通知** | Gmail からの予約系メールを検知した際の**このルーム向け通知**などに関わる設定。 |

モーダル下部の注記: **資料ライブラリの閲覧権限**はルームではなく、**メディア閲覧画面で資料ごと**に設定します。

### 4.5 新規ルーム追加時の設定モーダル

ルーム設定モーダルと同種のチェックに加え、**最終回: 継承 / 各回独立 / 1日まとめ**を選びます。  
「適用」で**次回のルーム追加時の既定値**として LocalStorage に保存される項目があります（AI会話返信の既定など）。

## 5. Line Summary Console — ユーザー権限（LINE ユーザー単位）

友だち追加・メッセージ送信のある **LINE ユーザーごと**にカードが並びます。

**並び順**: まず **役職**（`assigned_job_title`）が **役職ドロップダウンと同じ定義順**（代表取締役 → … → 一般社員）。未選択はその後、定義にない値はさらに後ろ。**同じ役職**の複数行は、**表示名のあいうえお順**（日本語の読みに近い `Intl.Collator('ja-JP')`、表示名が空のときは LINE ユーザー ID で比較）。表示名が同じときは LINE ユーザー ID で最終的に揃えます。

### 5.1 カード上部

| UI | 説明 |
|----|------|
| **表示名** | LINE 上の表示名。 |
| **ID表示** | LINE ユーザー ID をモーダル表示（コピー用）。 |

### 5.2 チェック（グリッド）

| チェック | 説明 |
|----------|------|
| **利用可** | このユーザーを **Bot 利用アクティブ**とみなすか。OFF だと権限メッセージ等の扱いになります（`is_active`）。 |
| **会話検索** | **過去メッセージ検索**機能を使えるか（`can_message_search`）。ルーム側の検索可否・除外ルームと組み合わせて最終判定されます。 |
| **資料検索** | **資料ライブラリ（アップロード資料）**を検索対象にできるか（`can_library_search`）。 |
| **予定閲覧** | カレンダー**一覧・確認**（「予定確認」「5月の会議は」など、予定の検索・閲覧）を許可するか（`can_calendar_view`）。**予定作成・予定更新とは独立**です。 |
| **予定作成** | カレンダー**新規作成**操作を許可するか（`can_calendar_create`）。 |
| **予定更新** | カレンダー**変更**（編集・削除に相当する操作）を許可するか（`can_calendar_update`）。 |
| **メディア** | **メディア保存・メディア系機能**にアクセスできるか（`can_media_access`）。ルームの「LINE添付保存」とセットで効きます。 |

※ **LINE Bot 側の挙動（この画面には項目なし）**: 会話検索はサーバーで **段階的**に試行します。**通常**は過去 **約6か月（180日）**までを上限にし（保持がそれより長くても同様）、**「会話検索フル」「会話検索全履歴」「会話検索裏」**のように指定したときだけ、**メッセージ保持設定の範囲いっぱい**まで段階を延ばします。**早い段階でキーワードにヒットしたとき**はそこで打ち切り、まだ古い帯が未走査なら Bot が **「さらに古い帯を検索するか」** を案内します（保留は `message_search_expand_pending_confirmations`。**会話検索 ON** かつルーム／除外設定の範囲内）。**ヒット 0 件のときの資料ライブラリ検索**（別テーブル）とは別フローです。

### 5.3 ルーム選択（会話検索の対象）

| UI | 説明 |
|----|------|
| **ルーム選択** | モーダルを開き、**会話検索の対象にしたいルーム**にチェックします（チェックあり＝検索対象）。 |
| **バッジ（全ルーム対象 / 除外 N件）** | いずれかのルームを対象外にすると「除外 N 件」表示。バックエンドでは「検索から除外するルーム ID のリスト」として `excluded_message_search_room_ids` に保存されます（チェックを外したルームがここに入ります）。 |

※モーダルでは **チェック＝検索対象**。チェックを外したルームが `excluded_message_search_room_ids` に含まれます。

**サーバー側の扱い（会話検索）**: `line-webhook` の全ルーム横断検索では、この配列に入った `room_id` のメッセージを結果から除外します。つまり **チェックを付けたルームの履歴だけ**がキーワード検索の候補に残ります。現在のチャットルームだけを検索するモードでは、そのルームが除外リストに含まれると検索できず、上記の案内メッセージになります。

### 5.4 所属店舗

| UI | 説明 |
|----|------|
| **所属店舗** | MARUGO GROUP 運営店舗一覧から**ドロップダウンで選択**します（`assigned_store`）。「未選択」にすると値は空（NULL）です。 |

### 5.5 役職

| UI | 説明 |
|----|------|
| **役職** | 定義済みの役職一覧から**ドロップダウンで選択**します（`assigned_job_title`）。例: 代表取締役、取締役、総務、マネージャー、アシスタントマネージャー、店長、店長補佐、料理長、一般社員。「未選択」にすると値は空（NULL）です。 |

### 5.6 フッターボタン

| ボタン | 説明 |
|--------|------|
| **再読込** | ユーザー権限一覧をサーバーから取り直す。 |
| **既存ユーザー取込** | 過去データから権限行を補完するバックフィル操作。 |

### 5.7 LINE での会話検索（管理 UI 外・運用者向けメモ）

管理画面には項目はありませんが、利用者が LINE で送る**文言**で検索範囲が変わります（実装: `line-webhook`）。

| モード | 入力例 | 検索が届く範囲の目安（保持が長い場合） |
|--------|--------|----------------------------------------|
| **通常（3段）** | `会話検索 試飲会` | **第1段**: まず過去 180 日（保持が短い場合はその範囲）を一度に検索。**第2段**: 0 件で保持がより長ければ、**その場で**保持いっぱいまで会話を追加検索。**第3段**: 会話でも 0 件なら **資料ライブラリを検索するか** `はい` / `いいえ`。 |
| **フル（ショートカット）** | `会話検索フル 試飲会`、`会話検索全履歴 試飲会`、`会話検索裏 試飲会`、`会話検索フルスキャン 試飲会`、`フル会話検索 試飲会`（`トーク検索…` 等も同様） | **第1・2段を省略**し、保持日数設定の上限まで会話を段階検索。 |

- **自然文だけ**で Groq が会話検索に振った場合も、**第1段は 180 日相当まで**。保持いっぱいまでは第2段が自動で続くか、**明示のフル用フレーズ**で省略できる。
- 定数名の参考: 通常の上限日数 **`MESSAGE_SEARCH_NORMAL_MAX_DAYS`**（コード上 180）。詳細はリポジトリの `README.md` §8.0 / §8.1 を参照。

### 5.8 サーバ上のデータの持ち方と AI（Groq）費用（運用メモ）

この画面からは設定できませんが、バックエンドでは次のように **検索と LLM を分離**しており、**トークや資料の総量が増えても、検索のたびに全文が AI に送られることはありません**。

| 種類 | サーバでの持ち方 | 検索時 | Groq |
|------|------------------|--------|------|
| **LINE トーク履歴** | `line_messages` に **1 メッセージ 1 行**で蓄積 | 期間内の行を読み、**キーワードルール**で絞り込み | **検索処理では不使用**。要約は **ヒット行のみ**・件数・文字に上限 |
| **資料ライブラリ** | アップロード時に本文抽出し **`extracted_text`** に保存 | ファイル名＋本文に **キーワード照合**、抜粋はルール生成 | **資料検索の表示生成では不使用** |

- そのため **AI（Groq）の従量は、単に履歴文字数が増えたからと跳ね上がる構造にはなっていない**（意図判定・要約など別経路が主因）。詳細は **`README.md` §7.1 / §8.5**。

## 6. LINE Media Viewer（`media.html`）

メディアファイルと資料ライブラリの**閲覧・削除・アップロード・権限**を行います。

### 6.1 接続・フィルタ（メディア一覧）

| フィールド | 説明 |
|------------|------|
| **プロジェクト URL** | 接続先 Supabase（読み取り専用表示の場合あり）。 |
| **管理トークン** | `admin-api` 用トークン。 |
| **種別フィルタ** | 画像 / 動画 / 音声 などメディア種別。 |
| **ルームで絞り込み** | 特定ルームのメディアのみ表示。 |
| **表示件数** | 1ページあたりの件数。 |
| **表示 / 再読み込み** | フィルタ適用・一覧更新。 |
| **一括削除** | 1回目: 選択モード ON（各カードにチェック）。2回目: 選択したファイルを削除（確認で「デリート」入力）。 |
| **アップロード上限（MB）** | メディア1件あたりの上限。**上限を保存**でサーバー側設定を更新。 |

### 6.2 メディア各カード

| UI | 説明 |
|----|------|
| **プレビュー** | 画像・動画など。 |
| **選択チェック（一括削除時）** | 削除対象に含めるか。 |
| **ダウンロード** | 可能な場合、ファイル取得。 |
| **削除** | 単体削除（確認で「デリート」入力）。 |

### 6.3 資料ライブラリ（アップロード・一覧）

| フィールド | 説明 |
|------------|------|
| **資料の紐付けルーム** | アップロード資料を論理的に紐づけるルーム。 |
| **ファイル選択 / ドラッグ＆ドロップ** | TXT / PDF / Word / Excel 等。 |
| **資料をアップロード** | Storage + メタデータ登録。 |
| **資料ルームで絞り込み / 表示件数** | 一覧のフィルタ。 |
| **表示 / 再読み込み** | 資料一覧の更新。 |

### 6.4 資料各行 — 閲覧権限

| 表示・操作 | 説明 |
|------------|------|
| **閲覧権限: 全ユーザー** | 許可ユーザー未設定。誰でも検索・参照可能な資料として扱われます。 |
| **閲覧権限: 指定ユーザーのみ（N人）** | 資料ごとに許可ユーザーを限定。**N は許可された LINE ユーザー数**。 |
| **閲覧設定** | モーダルを開き、**ユーザー名（表示名）のチェックボックス**で閲覧を許可するユーザーを選択。 |
| **全員一括選択** | クリックのたびに**全選択 ↔ 全解除**を切り替え。 |
| **保存** | `line_search_document_viewers` に反映。**1人もチェックしない場合は「全ユーザー閲覧」に戻る**扱い（実装メッセージに準拠）。 |
| **削除** | 資料メタとファイルの削除（確認で「デリート」入力）。 |

## 7. 用語対応（参考）

| UI ラベル | おおよその DB / API フィールド |
|-----------|-------------------------------|
| 配信を有効化 | `global_settings.is_enabled` 等 |
